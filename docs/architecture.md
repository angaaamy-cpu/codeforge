# معمارية CodeForge

## نظرة عامة

```
┌─────────────┐     ┌─────────────┐     ┌──────────────┐
│    Phone    │────▶│   GitHub    │────▶│  Codespaces  │
└─────────────┘     └─────────────┘     └──────────────┘
                                                 │
                                                 ▼
┌─────────────┐     ┌─────────────────┐     ┌──────────────┐
│     Git     │◀────│ Orchestration   │◀────│  Workspace   │
└─────────────┘     │    Layer        │     └──────────────┘
                    └─────────────────┘
```

## تدفق البيانات

1. **الطلب**: المستخدم يرسل طلباً من الهاتف
2. **المصادقة**: GitHub يتحقق من الهوية
3. **التنفيذ**: Codespaces يستقبل التعليمات
4. **التنسيق**: طبقة التنسيق تدير الوكلاء
5. **العمل**: Workspace ينفذ المهام
6. **الحفظ**: Git يسجل التغييرات

## الطبقات المعمارية

### طبقة التقديم (Presentation Layer)
- **الواجهة**: تطبيق الهاتف أو الويب
- **البروتوكول**: REST API / GraphQL
- **المصادقة**: OAuth2 عبر GitHub

### طبقة التنسيق (Orchestration Layer) ⭐
- **التقنية**: CrewAI (مرحلة 2)
- **المسؤولية**: تنسيق الوكلاء وتوزيع المهام
- **الحالة**: مخطط - سيُضاف لاحقاً

### طبقة التنفيذ (Execution Layer)
- **التقنية**: OpenHands (مرحلة 3)
- **المسؤولية**: تنفيذ الكود والتفاعل مع النظام
- **الحالة**: مخطط - سيُضاف لاحقاً

### طبقة الذاكرة (Memory Layer)
- **الآن**: `docs/` - ملفات Markdown منظمة
- **المستقبل**: ChromaDB للذاكرة المتجهة
- **الانتقال**: ADR مخصص عند الانتقال

### طبقة التخزين (Storage Layer)
- **الكود**: GitHub Repository
- **التوثيق**: `docs/` في المستودع
- **البيانات**: Git Issues + Discussions

## الوكلاء (Agents)

### 1. Manager Agent
- تنسيق المهام
- تتبع التقدم
- إدارة الأولويات

### 2. Architect Agent
- تحليل المتطلبات
- تصميم المعمارية
- كتابة ADR

### 3. Developer Agent
- كتابة الكود
- تنفيذ المهام
- تطبيق المعايير

### 4. QA Agent
- اختبار الكود
- كشف الأخطاء
- التحقق من الجودة

## الامتدادات المستقبلية

### ChromaDB Integration
```
User Query → Embedding → ChromaDB → Relevant Docs → Response
```

### Multi-Model Support
- Gemini (مرحلة 2)
- Claude (مرحلة 4)
- GPT-4 (مرحلة 4)

## القيود المعمارية

- لا Zustand أو Redux (حالة بسيطة)
- لا GraphQL subscriptions (مرحلة 4)
- لا microservices ( monolithic أولاً)
