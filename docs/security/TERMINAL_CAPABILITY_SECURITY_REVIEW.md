# TERMINAL_CAPABILITY_SECURITY_REVIEW.md

**السياق**: `docs/adr/013-src-core-architectural-status.md` قرَّر أن قدرة `"terminal"` المُسجَّلة في `src/Core/capability.py` تبقى **ميتاداتا فقط بلا أي handler حقيقي** حتى مراجعة أمنية منفصلة. هذا هو تلك المراجعة. **لا تنفيذ هنا** — هذا مستند قرار فقط.

---

## 1. الوضع الحالي (الحقيقة، لا الافتراض)

```python
# src/Core/capability.py - السطر الفعلي المسجَّل حالياً
terminal_cap = Capability(
    name="terminal",
    description="Execute shell commands",
    permissions=[Permission.EXECUTE],
    tags=["core", "system"],
)
registry.register(terminal_cap)   # tools = {} — فارغ تماماً
```
**لا يوجد أي كود اليوم** يُنفِّذ أمر shell واحداً حقيقياً في أي مسار قابل للوصول من التطبيق. `Capability.tools` فارغ. هذا هو خط الأساس الآمن الحالي.

## 2. لماذا هذا خطر حقيقي إن نُفِّذ بسذاجة (Threat Model)

| التهديد | كيف يحدث لو رُبِط handler ساذج (`subprocess.run(user_input)`) |
|---|---|
| **Remote Code Execution** | أي مستخدم يصل لـ `/api/*` (حتى مع مصادقة Phase 1 — لكنها admin key واحد لكل النظام، لا عزل مستخدمين) يستطيع تنفيذ أي أمر على السيرفر المُستضيف |
| **Command Injection** | حتى مع "قائمة أوامر مسموحة"، أي تمرير غير مُطهَّر لمعاملات (arguments) يفتح باب حقن (`; rm -rf`, `$(...)`, backticks) |
| **Secrets Exfiltration** | `SecretsManager` (`src/Core/secrets.py`) يعيش **في نفس عملية Python** — أمر shell بصلاحية القراءة يمكنه قراءة متغيرات البيئة (`GEMINI_API_KEY`, `ADMIN_API_KEY`, `SECRET_KEY`) مباشرة من `/proc/self/environ` أو الذاكرة |
| **Path Traversal / Workspace Escape** | تنفيذ Shell لا يلتزم تلقائياً بحدود `PathService.is_within_root()` — قيد أمان المسارات (SYSTEM_INVARIANTS #3) **لا ينطبق على أوامر Shell إطلاقاً**، فهو مصمم لعمليات ملفات Python فقط |
| **Resource Exhaustion / DoS** | أمر مثل `:(){ :|:& };:` أو حلقة لا نهائية يعطِّل الـ worker (وربما العملية بأكملها تحت `--workers 2` المحدودين) |
| **Persistence** | أمر shell قادر على كتابة cron job أو تعديل ملفات النظام خارج نطاق `WORKSPACE_ROOT` بالكامل — لا علاقة له بقيود PathService |
| **Lateral Movement** | إن كانت بيئة النشر (Railway/Render) تشارك شبكة داخلية، تنفيذ Shell يفتح باباً لاستكشاف الشبكة الداخلية |

**الخلاصة**: قدرة Terminal الحقيقية = مساحة هجومية تتجاوز كل ضوابط الأمان الأخرى المبنية في هذا المشروع (Phase 1 auth، PathService sandbox) لأنها تعمل **تحت** مستوى تلك الضوابط، لا فوقها.

## 3. الخيارات المتاحة (من الأخطر للأكثر أماناً)

| الخيار | الوصف | التوصية |
|---|---|---|
| **A. Shell عام غير مقيَّد** (`subprocess.run(cmd, shell=True)`) | أي أمر، أي معامل | **مرفوض قطعياً** - لا مبرر يوازي المخاطر |
| **B. Allowlist أوامر ثابتة + معاملات مُطهَّرة** (مثال: `npm install`, `git status` فقط، بمعاملات مفحوصة بـ regex صارم) | نطاق محدود جداً، كل أمر مُراجَع يدوياً مسبقاً | **الأقرب للقبول** - لكن يحتاج تصميم منفصل لكل أمر مسموح + توقيع صريح منك على القائمة |
| **C. Sandboxing كامل** (container منفصل معزول شبكياً، لا صلاحية كتابة خارج مجلد مؤقت، بلا وصول لمتغيرات البيئة الحساسة، timeout صارم) | يقلل الأثر حتى لو استُغِلت ثغرة | يحتاج بنية تحتية غير متوفرة حالياً (لا Docker-in-Docker موثَّق في المشروع) |
| **D. البقاء ميتاداتا فقط (الوضع الحالي)** | لا تنفيذ إطلاقاً | **التوصية الحالية** حتى تتوفر البنية التحتية لـ C أو تُحدَّد قائمة B بدقة صريحة منك |

## 4. متطلبات إلزامية *إن* تقرَّر لاحقاً تنفيذ أي شيء أبعد من (D)

1. **Allowlist صريح مكتوب** لكل أمر مسموح (لا "أي أمر git" بل "git status، git log --oneline -20" تحديداً بمعاملاتها).
2. **Sandbox process معزول** لا يرث متغيرات البيئة الحساسة (`env={}` صريح، لا `os.environ` كاملاً).
3. **Timeout صارم** (`subprocess.run(..., timeout=N)`) على كل استدعاء.
4. **Audit log إلزامي** لكل أمر مُنفَّذ فعلياً (من نفَّذه، متى، ماذا، النتيجة) - غير قابل للحذف من نفس العملية.
5. **حدود موارد** (CPU/Memory/عدد العمليات الفرعية) عبر `resource` module أو cgroups.
6. **لا تنفيذ داخل نفس عملية Python** التي تحمل الأسرار الحية (`SecretsManager` في الذاكرة) - فصل عملية كامل كحد أدنى.
7. **مراجعة أمنية منفصلة مرة أخرى للتصميم الفعلي** (هذا المستند تقييم مخاطر عام، ليس مراجعة تصميم لتطبيق فعلي لم يُكتَب بعد).

## 5. التوصية النهائية

**لا تنفيذ لقدرة Terminal الآن.** تبقى `docs/adr/013-src-core-architectural-status.md` سارية: تسجيل ميتاداتا فقط. Phase 5 (إن استُؤنِفت) تقتصر على **files/git فقط** عبر أدابتر يربط `Capability.tools` بالكود الإنتاجي الموجود فعلاً (`path_service.py`, `git_manager.py`) — كلاهما مقيَّد أصلاً بضوابط أمان موجودة (PathService sandbox، GIT_AUTO_PUSH=False)، بخلاف Terminal الذي لا ضابط أمان بنيوي له إطلاقاً.

**Browser** (المذكورة أيضاً في Phase 5 الأصلية) لم تُقيَّم هنا لأنها غير موجودة إطلاقاً حتى كتسجيل ميتاداتا (بخلاف Terminal) — تحتاج مراجعة مخاطر منفصلة كاملة إن طُرِحت مستقبلاً (SSRF، تسريب كوكيز/جلسات، الوصول لشبكات داخلية عبر طلبات الخادم).
